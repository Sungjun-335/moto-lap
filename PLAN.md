# 트랙 DB 기능 계획서

## 현재 상태 요약

**프로젝트**: motolap - 모터사이클 랩 분석 웹 앱
**기술 스택**: React + TypeScript + Vite (프론트엔드) / Cloudflare Workers + D1 (백엔드)
**지도**: Google Maps API (`@react-google-maps/api`)

### 현재 동작 방식
1. 사용자가 AIM 데이터 로거의 CSV 파일을 업로드합니다
2. CSV에서 GPS 좌표(위도/경도), 속도, G값 등의 데이터를 추출합니다
3. 비콘 마커(Beacon Marker) 기준으로 랩을 나눕니다
   - 비콘 마커는 트랙에 설치된 자석 센서를 통과할 때 기록되는 타임스탬프입니다
   - 마치 "출발선을 지날 때마다 시간을 찍는 것"과 같습니다
4. 각 랩에서 **속도 변화 + 횡가속도**를 분석하여 코너를 자동으로 감지합니다
5. 감지된 코너에 T1, T2, T3... 같은 번호가 붙습니다

### 현재 문제점
- 코너 번호가 세션마다 달라질 수 있습니다 (T1이 항상 같은 코너를 가리키지 않음)
- 코너에 이름이 없습니다 ("1번 코너", "헤어핀" 같은 실제 이름)
- 서킷 정보가 저장되지 않아서, 같은 트랙에서 여러 번 주행해도 매번 새로 감지합니다
- 다른 사람의 데이터와 비교할 때 코너 기준이 통일되지 않습니다

### 사용 가능한 데이터 (TRP CSV 예시)
| 항목 | CSV 컬럼 | 설명 |
|------|----------|------|
| GPS 위치 | `GPS_Latitude`, `GPS_Longitude` | 위도/경도 좌표 (약 50Hz로 기록) |
| GPS 속도 | `GPS_Speed` | km/h 단위 속도 |
| 횡가속도 | `LateralAcc` (`GPS_LatAcc`) | 좌우로 쏠리는 힘 (g 단위) |
| 종가속도 | `InlineAcc` (`GPS_LonAcc`) | 가속/감속 힘 (g 단위) |
| GPS 방향 | `GPS_Heading` | 진행 방향 (도 단위) |
| RPM | `RPM` | 엔진 회전수 |
| TPS | `TPS` | 스로틀 열림 정도 (%) |
| 기어 | `Gear` | 현재 기어 단수 |

- TRP 위치: 위도 약 37.0827도, 경도 약 129.0182도 (강원도 태백)
- 샘플레이트: 50Hz (초당 50개 데이터)
- 1랩 약 63-64초, 총 10랩 세션

## 최종 목표

**"서킷별로 코너 정보가 미리 저장되어 있어서, CSV를 올리면 어느 서킷인지 자동으로 인식하고, 코너마다 정해진 이름과 번호가 일관되게 붙는 시스템"**

구체적으로:
- 태백 레이싱파크(TRP)의 트랙 형상, 코너 위치, 코너 이름이 DB에 저장됩니다
- 새 CSV를 업로드하면 GPS 좌표를 보고 "이건 TRP 데이터구나" 하고 자동으로 인식합니다
- T1은 항상 같은 코너를 가리키며, "헤어핀", "S커브" 같은 이름이 표시됩니다
- 이후 다른 서킷(영암, 인제 등)도 같은 방식으로 추가할 수 있습니다

### Phase 0: TRP 트랙 데이터 수동 제작

**목표**: 실제 GPS 주행 데이터와 위성 사진을 합쳐서 TRP의 "공식 트랙 지도"를 만듭니다.

이 단계가 전체 작업의 **기초 공사**입니다. 건물을 짓기 전에 설계도를 그리는 것과 같습니다.

#### 0-1. GPS 경로에서 트랙 중심선 추출

**할 일**: 이미 있는 CSV 파일(`tan ga_TRP_a_0498.csv`)에서 GPS 좌표만 뽑아서 "트랙이 어떤 모양인지" 파악합니다.

**구체적 작업**:
1. CSV에서 한 랩 분량의 GPS_Latitude, GPS_Longitude를 추출합니다
2. 여러 랩의 GPS 경로를 겹쳐서 **평균 경로(중심선)**를 계산합니다
   - 한 바퀴만 쓰면 주행 라인에 따라 약간 치우칠 수 있으므로,
     여러 바퀴를 평균 내면 트랙의 "중앙선"에 가까운 경로가 나옵니다
3. 이 중심선을 일정 간격(예: 1미터마다)으로 샘플링하여 저장합니다

**왜 중요한가**: 이 중심선이 "트랙의 뼈대"가 됩니다. 나중에 모든 코너 위치, 구간 거리 등이 이 중심선을 기준으로 계산됩니다.

**완료 확인**: 구글 맵에 추출한 중심선을 그려봤을 때, 위성 사진의 트랙 모양과 잘 맞아야 합니다.

**예상 소요 시간**: 2-3시간

#### 0-2. 위성 사진 참조하여 코너 확정

**할 일**: 구글 맵 위성 사진을 보면서 GPS 경로 위에 코너 위치와 이름을 지정합니다.

**구체적 작업**:
1. 구글 맵에서 TRP를 위성 모드로 봅니다 (좌표: 37.0827, 129.0182 부근)
2. GPS 중심선을 지도 위에 그립니다
3. 코너의 **시작점(진입)**, **꼭짓점(에이펙스)**, **끝점(탈출)**을 중심선 상의 거리(distance)로 기록합니다
4. 각 코너에 다음 정보를 부여합니다:
   - **번호**: T1, T2, T3... (출발선부터 순서대로)
   - **이름**: 공식 이름 또는 라이더들이 부르는 이름 (예: "1코너", "헤어핀", "S구간")
   - **방향**: L(좌회전) 또는 R(우회전)
   - **구간 타입**: hairpin(급커브), sweeper(완만한 커브), chicane(S자), kink(미세한 꺾임) 등

**왜 이 단계를 사람이 해야 하는가**:
- 자동 코너 감지는 "속도가 떨어지는 곳 = 코너"로 판단하지만, 실제로는 라이더마다 다르게 해석합니다
- 위성 사진과 실제 트랙 지식을 결합해야 정확한 코너 구분이 가능합니다
- 예를 들어, 연속된 두 코너를 하나의 "S커브"로 볼 것인지, 별도의 T3/T4로 볼 것인지는 사람의 판단이 필요합니다

**주의사항**:
- TRP에 직접 가서 주행해본 경험이 있으면 훨씬 정확합니다
- 위성 사진에서 노면 표시(커브스톤, 페인트 라인)를 참고하면 좋습니다

**완료 확인**: 각 코너의 위도/경도, 중심선 거리, 이름, 방향이 모두 기록된 목록이 완성되어야 합니다.

**예상 소요 시간**: 2-4시간 (트랙 지식에 따라 다름)

### Phase 1: 트랙 데이터 구조 설계 & JSON 파일 작성

**목표**: Phase 0에서 만든 트랙 정보를 코드에서 사용할 수 있는 형식으로 저장합니다.

#### 1-1. 트랙 데이터 타입 정의

**할 일**: TypeScript에서 트랙 정보를 담을 데이터 구조를 설계합니다.

**저장할 정보 (Track 타입)**:
```
Track (서킷 정보)
  - id: 고유 식별자 (예: "trp")
  - name: 서킷 이름 (예: "태백 레이싱파크")
  - shortName: 줄임말 (예: "TRP")
  - country: 국가 코드 (예: "KR")
  - location: { lat, lon } - 서킷 중심 좌표
  - length: 트랙 총 길이 (미터)
  - direction: "CW" 또는 "CCW" (시계/반시계 방향)
  - centerline: [ {lat, lon, dist} ... ] - 중심선 좌표 배열
  - sectors: [ {id, name, startDist, endDist} ] - 구간 정보
  - corners: [ TrackCorner ... ] - 코너 목록
```

**저장할 정보 (TrackCorner 타입)**:
```
TrackCorner (코너 정보)
  - id: 코너 번호 (1, 2, 3...)
  - name: 코너 이름 (예: "헤어핀")
  - direction: "L" 또는 "R"
  - type: "hairpin" | "sweeper" | "chicane" | "kink"
  - apex: { lat, lon, dist } - 꼭짓점 위치
  - entry: { lat, lon, dist } - 진입점 위치
  - exit: { lat, lon, dist } - 탈출점 위치
  - idealGear: (선택) 이상적인 기어 단수
  - notes: (선택) 참고 메모
```

여기서 `dist`는 "출발선에서부터의 거리 (미터)"입니다. 이것이 핵심 기준이 됩니다.
GPS 좌표는 위치를 보여주기 위한 것이고, 실제 매칭은 거리(dist) 기준으로 합니다.

**파일 위치**: `frontend/src/types.ts`에 타입을 추가합니다.

**예상 소요 시간**: 1-2시간

**완료 확인**: JSON 파일이 TypeScript 타입에 맞게 작성되고, 에러 없이 import 됩니다.

### Phase 2: 세션 업로드 시 트랙 자동 인식

**목표**: CSV 파일을 올렸을 때, GPS 좌표를 보고 "이 세션은 TRP에서 기록되었다"고 자동으로 판단합니다.

#### 2-1. 트랙 매칭 알고리즘 구현

**할 일**: GPS 좌표와 트랙 DB를 비교하여 어느 서킷인지 판별하는 함수를 만듭니다.

**동작 원리** (쉽게 설명하면):
1. CSV에서 처음 몇 초 동안의 GPS 좌표를 읽습니다
2. 이 좌표들의 "중심점"을 계산합니다 (모든 좌표의 평균)
3. DB에 저장된 각 서킷의 중심 좌표와 비교합니다
4. 가장 가까운 서킷이 1km 이내이면 "매칭 성공"으로 판단합니다

이것은 마치 "내 위치에서 가장 가까운 서킷을 찾는 것"과 같습니다.
더 정밀한 검증이 필요하면, GPS 경로의 전체 모양과 저장된 중심선의 모양을 비교할 수도 있습니다.

**알고리즘 상세**:
```
1단계: 조기 필터 (빠른 검사)
  - 세션 GPS 중심점 계산
  - 각 트랙 center와의 거리 계산 (Haversine 공식)
  - 5km 이내인 트랙만 후보로 남김

2단계: 정밀 매칭 (후보가 여러 개일 때)
  - 세션 GPS 포인트들을 트랙 중심선과 비교
  - 각 포인트에서 중심선까지의 평균 거리가 가장 작은 트랙 선택
  - 평균 거리가 50m 이내이면 매칭 성공
```

**파일 위치**: `frontend/src/utils/trackMatcher.ts`

**완료 확인**: TRP CSV를 업로드했을 때 "태백 레이싱파크" 로 자동 인식됩니다.

**예상 소요 시간**: 3-4시간

#### 2-2. 파서에 트랙 매칭 연동

**할 일**: 기존 CSV 파싱 과정에 트랙 매칭을 추가합니다.

**수정 대상 파일**: `frontend/src/components/FileUpload.tsx`

**변경 내용**:
1. CSV 파싱이 끝나면 `matchTrack(dataPoints)` 함수를 호출합니다
2. 매칭된 트랙 정보를 `SessionData`에 추가합니다
3. UI에 "감지된 서킷: 태백 레이싱파크 (TRP)" 메시지를 표시합니다
4. 매칭 실패 시 "서킷을 자동으로 인식하지 못했습니다. 수동 선택하시겠습니까?" 메시지를 표시합니다

**타입 변경**: `SessionData.metadata`에 `trackId?: string` 필드를 추가합니다.

**예상 소요 시간**: 2-3시간

### Phase 3: 코너 감지를 트랙 DB 기반으로 전환

**목표**: 기존의 "속도 기반 자동 감지"를 "트랙 DB의 코너 위치 기반 감지"로 전환합니다.

이 단계가 핵심입니다. 현재는 속도 변화를 보고 "여기가 코너인 것 같다"고 추측하지만,
트랙 DB가 있으면 "여기가 코너다"라고 확정적으로 알 수 있습니다.

#### 3-1. 거리 기반 코너 매핑 함수

**할 일**: 트랙 DB의 코너 구간(entry_dist ~ exit_dist)을 세션 GPS 데이터에 매핑합니다.

**동작 원리**:
1. 세션의 각 GPS 포인트에 대해, 트랙 중심선 위의 가장 가까운 점을 찾습니다
2. 그 점의 `dist` 값(출발선으로부터 거리)을 해당 GPS 포인트에 부여합니다
3. 이제 각 포인트가 "출발선으로부터 150m 지점"처럼 트랙 위치를 가집니다
4. 트랙 DB에서 "T1은 150m~220m"라고 되어 있으면, 해당 범위의 포인트들이 T1입니다

이것은 마치 "지도에서 현재 위치를 도로 위에 표시하는 것"과 같습니다.
GPS가 약간 흔들려도, 가장 가까운 중심선 위의 점을 기준으로 하므로 안정적입니다.

**파일 위치**: `frontend/src/utils/trackProjection.ts`

**수정 대상**: `cornerDetection.ts`에 "트랙 DB가 있으면 DB 기반, 없으면 기존 자동 감지" 분기를 추가합니다.

**예상 소요 시간**: 4-5시간

#### 3-2. 하이브리드 감지 모드

**할 일**: 트랙 DB 기반 코너 위치 + 기존 속도 기반 상세 메트릭을 결합합니다.

**왜 필요한가**:
- 트랙 DB: "여기가 T3 코너이고, 이름은 헤어핀이다" (위치와 이름을 확정)
- 속도 분석: "이 T3에서 진입속도 85km/h, 최저속도 42km/h" (성능 수치를 계산)
- 두 가지를 합치면 "T3 헤어핀: 진입 85, 최저 42, 탈출 78" 같은 완전한 정보가 됩니다

**동작 방식**:
1. 트랙 DB에서 코너 구간(거리 범위)을 가져옵니다
2. 해당 구간의 GPS 데이터 인덱스를 찾습니다
3. 그 인덱스 범위에서 기존 `detectCorners`의 메트릭 계산 로직을 적용합니다
4. 결과를 합쳐서 "이름 있는 코너 + 정밀 메트릭"을 반환합니다

**수정 대상**: `cornerDetection.ts`, `types.ts` (Corner 타입에 `trackCornerId`, `name` 필드 추가)

**예상 소요 시간**: 3-4시간

## 핵심 기술 결정 사항

### Q: 트랙 데이터를 JSON 파일로 저장할까, DB에 저장할까?

**초기 결정: JSON 파일** (Phase 1~4)

| 기준 | JSON 파일 | D1 데이터베이스 |
|------|-----------|-----------------|
| 구현 난이도 | 쉬움 | 중간 |
| 서킷 추가 | 파일 추가 + 배포 | API 호출 |
| 오프라인 동작 | 가능 (번들에 포함) | 불가능 |
| 수정 권한 관리 | Git으로 관리 | 인증 시스템 필요 |
| 적합한 시기 | 서킷 5개 이하일 때 | 사용자가 직접 추가해야 할 때 |

JSON 파일은 빌드할 때 앱에 포함되므로, 인터넷 연결 없이도 트랙 인식이 됩니다.
한국 서킷 수가 10개 미만이므로, 상당 기간 JSON으로 충분합니다.

### Q: GPS 좌표 비교를 위한 거리 계산은 어떻게?

**Haversine 공식** 사용:
- 두 GPS 좌표 사이의 실제 거리(미터)를 계산하는 공식입니다
- "지구가 둥글다"는 것을 고려한 정밀 계산입니다
- 서킷 규모(~2km)에서는 오차가 무시할 만합니다

### Q: 중심선 좌표는 몇 개 정도 저장하나?

- 1미터 간격으로 저장하면 TRP는 약 1,340개 포인트
- JSON 용량은 약 40~50KB (가벼움)
- 정밀도와 용량의 좋은 균형점입니다

## 전체 주의사항

1. **GPS 정밀도 한계**: GPS 좌표는 1~3미터 정도의 오차가 있습니다.
   코너 경계를 너무 촘촘하게 설정하면 오차 때문에 코너가 잘못 인식될 수 있습니다.
   코너 경계에 5~10미터 정도의 여유를 두세요.

2. **트랙 DB가 없는 서킷**: 트랙 DB에 없는 서킷의 CSV가 업로드될 수 있습니다.
   이때는 기존 자동 감지 방식으로 동작해야 합니다 (하위 호환성).
   "알 수 없는 서킷 - 자동 감지 모드" 같은 메시지를 표시합니다.

3. **중심선 정렬**: GPS 데이터의 출발선 위치가 세션마다 약간 다를 수 있습니다.
   중심선의 시작점(dist=0)과 실제 데이터의 시작점을 정렬하는 로직이 필요합니다.
   비콘 마커(랩 시작 지점)를 기준으로 정렬하면 좋습니다.

4. **역주행 감지**: 드물지만 역방향으로 주행하는 데이터가 올 수 있습니다.
   GPS Heading(진행 방향) 값을 확인하여 트랙 방향과 일치하는지 검증하면 좋습니다.

5. **기존 기능 보존**: Phase 3에서 코너 감지 로직을 변경할 때, 기존 자동 감지가 깨지지 않도록 주의합니다. 트랙 DB가 있으면 DB 기반, 없으면 기존 방식 - 이 분기가 확실해야 합니다.

## 문제가 생겼을 때

### "GPS 좌표가 트랙 밖을 가리킨다"
- GPS 오차 때문에 일부 포인트가 트랙 밖으로 벗어날 수 있습니다
- 해결: 중심선에서 50m 이상 떨어진 포인트는 무시하거나, 가장 가까운 중심선 점으로 보정합니다

### "서킷 인식이 안 된다"
- 서킷 중심 좌표와 실제 GPS 좌표가 너무 다를 수 있습니다
- 해결: 트랙 JSON의 `location` 좌표가 정확한지 확인합니다
- 매칭 거리 임계값(5km)을 늘려서 테스트합니다

### "코너 번호가 하나씩 밀린다"
- 출발선에서의 시작 위치 오프셋이 맞지 않을 수 있습니다
- 해결: 비콘 마커를 기준으로 거리를 재정렬합니다

### "새 서킷을 추가하고 싶다"
- Phase 0의 과정(GPS 추출 + 위성 사진 참조)을 반복합니다
- JSON 파일을 `frontend/src/data/tracks/` 폴더에 추가합니다
- `frontend/src/data/tracks/index.ts`에 import를 추가합니다
